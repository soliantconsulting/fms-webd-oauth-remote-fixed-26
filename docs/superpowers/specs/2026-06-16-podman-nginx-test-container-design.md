# Podman + nginx test container for local HTTPS serving

**Date:** 2026-06-16
**Status:** Approved (design)

## Background

This repo is a static HTML/JS demo of starting the FileMaker WebDirect OAuth login flow
from a web site hosted on a **separate** web server (the "remote" scenario). The OAuth port
(see [2026-06-16-oauth-config-and-feature-flag-design.md](2026-06-16-oauth-config-and-feature-flag-design.md))
is implemented but its **Task 8 manual end-to-end verification** is blocked: the flow needs
the page served over **HTTPS from a host matching `webDNS`** (`webdlogin.ets.fm`).

This spec covers a throwaway local container — Ubuntu + nginx, run with Podman — that serves
the repo's static files over HTTPS on the configured `webDNS` hostname, so both OAuth modes
(popup and full-page redirect) can be exercised against a real FileMaker Server + identity
provider. It is a **test harness only**; it changes no application code.

## Why HTTPS-on-webDNS specifically (the constraint that shapes the design)

Tracing who connects to `webDNS` in the flow:

- The **identity provider's** registered redirect URI points at **FileMaker Server**
  (`fmsDNS`), not at `webDNS`. The IdP redirects the browser back to FMS's OAuth callback;
  it never connects to `webDNS` and never evaluates its certificate. `fmsDNS` keeps its own
  real, publicly-trusted certificate, untouched by anything here.
- After FMS finishes the handshake it sends the **browser** an HTTP `302` to
  `X-FMS-Return-URL` (`https://<webDNS>/oauth-landing.html` in popup mode, the index URL in
  full-page mode). FMS does not make a server-to-server call to `webDNS`; it hands the
  browser a `Location:` header and the **browser** follows it.

So the **only** party that ever connects to `webDNS` and evaluates its certificate is the
**user's browser** — a human who can accept a one-time warning. This is why a runtime
self-signed certificate for `webDNS` is sufficient. Plain HTTP would not work, for a
different reason: the application code emits `https://` return URLs and the popup's
same-origin `storage` event requires matching secure origins.

## Decisions

- **Base image:** `docker.io/library/ubuntu:24.04` (LTS), nginx + openssl installed.
- **TLS:** runtime self-signed certificate generated by the entrypoint from a `WEBDNS`
  environment variable (so it matches whatever hostname is configured — the hostname is a
  variable, never baked into the image). A user-supplied certificate may be mounted to skip
  generation.
- **File serving:** volume-mount the repo's web files into the nginx webroot (read-only) so
  edits show on browser refresh with no rebuild.
- **Hostname resolution:** documented `/etc/hosts` mapping (`127.0.0.1 webdlogin.ets.fm`),
  keeping the cert hostname and the browser URL matching as the OAuth origin checks require.

## Components

All new, under a `container/` directory. None modify application code.

| File | Responsibility |
|------|----------------|
| `container/Containerfile` | `FROM docker.io/library/ubuntu:24.04`; `apt-get install` nginx + openssl (then clean apt lists); copy `nginx.conf.template` and `entrypoint.sh`; `EXPOSE 443`; `ENTRYPOINT ["/entrypoint.sh"]`. |
| `container/entrypoint.sh` | Bash, `set -e`. Read `WEBDNS` (default `webdlogin.ets.fm`). If `/etc/nginx/certs/server.crt` + `server.key` are not already present (mounted), generate a self-signed cert+key into `/etc/nginx/certs` with `openssl req -x509 -newkey rsa:2048 -nodes -days 825` and `-subj "/CN=${WEBDNS}"` + `-addext "subjectAltName=DNS:${WEBDNS}"`. Render the nginx conf by substituting `${WEBDNS}` into the template (via `envsubst` or `sed`). `exec nginx -g 'daemon off;'`. |
| `container/nginx.conf.template` | One `server` block: `listen 443 ssl;`, `server_name ${WEBDNS};`, `root /usr/share/nginx/html;`, `index index.html;`, `ssl_certificate /etc/nginx/certs/server.crt;`, `ssl_certificate_key /etc/nginx/certs/server.key;`. Adds `add_header Cache-Control "no-store";` so edits to mounted files appear on refresh. |
| `container/run.sh` | Bash convenience wrapper. `podman build -t fms-webd-oauth-test -f container/Containerfile .` then `podman run --rm -it -p 443:443 -v "<repo-root>:/usr/share/nginx/html:ro,Z" -e WEBDNS="${WEBDNS:-webdlogin.ets.fm}" fms-webd-oauth-test`. Echoes the `/etc/hosts` reminder and the `https://<WEBDNS>` URL. Repo root derived from the script's own location. |

## How it runs (data flow)

1. `container/run.sh` builds the image and runs it: repo mounted read-only at the webroot,
   port 443 published to the host, `WEBDNS` passed in (defaulting to `webdlogin.ets.fm`).
2. The entrypoint generates a self-signed cert for `WEBDNS` (unless a cert is mounted at
   `/etc/nginx/certs`), renders the nginx config, and starts nginx in the foreground.
3. The tester adds `127.0.0.1 webdlogin.ets.fm` to `/etc/hosts` (one time), browses to
   `https://webdlogin.ets.fm`, and accepts the one-time browser certificate warning.
4. nginx serves the live repo files. Editing `assets/js/oauth-config.js` (or any file) plus
   a browser refresh reflects immediately — no rebuild — thanks to the read-only mount and
   `Cache-Control: no-store`.

## README addition

A new "Local testing with Podman" section in `README.txt` covering:

- **Build & run:** `container/run.sh` (and the equivalent raw `podman build`/`podman run`).
- **Hostname:** add `127.0.0.1 webdlogin.ets.fm` to `/etc/hosts` (or use real DNS), and keep
  `webDNS` in `oauth-config.js` matching it.
- **SSL explanation (explicitly requested):** the cert is self-signed and generated at run
  time for the `WEBDNS` you pass. Only the **browser** ever sees this cert — the IdP
  redirects to FMS, and FMS returns a `302` to the browser — so neither the IdP nor FMS
  validates it, and accepting the one-time browser warning is safe. To avoid the warning,
  mount a real or locally-trusted cert at `/etc/nginx/certs` (`server.crt` / `server.key`).

## Out of scope

- Production deployment, hardening, or non-test serving.
- Auto-installing/trusting the self-signed CA in the host trust store (the warning is
  accepted manually; a mounted cert is the documented alternative).
- CI integration.
- Any change to application code, `oauth-config.js` values, or the OAuth logic.

## Relationship to other work

- Unblocks **Task 8** (manual end-to-end verification) of the OAuth config + feature-flag
  port, which requires HTTPS serving on `webDNS`.
- Lives on the same branch (`feature/oauth-config-and-feature-flag`) per the agreed

## Success criteria

- `container/run.sh` builds the Ubuntu+nginx image and runs it with no manual cert steps.
- Browsing to `https://<WEBDNS>` serves `index.html` over HTTPS with a cert whose SAN matches
  `WEBDNS` (one-time browser warning only).
- Editing a mounted repo file and refreshing the browser shows the change without a rebuild.
- Passing a different `WEBDNS` produces a cert and `server_name` for that hostname.
- A cert mounted at `/etc/nginx/certs` is used instead of generating one.
- `README.txt` documents build/run, the `/etc/hosts` mapping, and the SSL explanation.
